摘要

教育部有关政策提出促进教务管理全流程数字型转型，毕业论文（设计）选题管理是助力人才培养质量提升的重要环节。围绕高校毕业论文选题所面临的“信息分散、匹配粗放、过程难追踪”共性问题，本文设计并实现了一套面向学生、导师和管理员协同的选题管理平台。系统采用 Spring Boot + Vue3 搭建前后端分离架构，以“学生—导师—选题”统一标签空间作为数据基础，让学生画像、导师画像与题目需求产生关联；在业务流程上，平台同时支持“先选题后审核”与“先选导师再安排题目”两种入口，并在后端合流为统一的申请实体，以保证流程灵活性与管理一致性。系统落地了三项关键能力：其一，基于规则映射与大模型抽取的标签生成机制；其二，融合“标签重合度、语义向量、导师画像相似度”的匹配算法，并在向量服务不可用时回退到标签侧得分以保障主流程连续；其三，“AI 生成候选题目—相似度去重—导师人工定稿”的命题协同机制。工程实现中，系统通过事务边界与数据库约束共同保证状态写入的完整性，同时提供面向管理员的运行监控与日志查看接口，以支持异常场景下的定位与治理。该平台能够降低选题沟通与筛选成本，改善学生与题目以及与导师之间的匹配效率，并为后续质量评估与题库改进提供持续的数据链路。

关键词：毕业论文选题；导师管理；标签化建模；智能推荐；协同管理系统


Abstract

The Ministry of Education has proposed policies to promote the full-process digital transformation of educational administration management, with graduation thesis (design) topic selection management being an important link to improve the quality of talent cultivation. Addressing the common problems of "information dispersion, coarse-grained matching, and difficult process tracking" in university graduation thesis topic selection, this paper designs and implements a collaborative topic selection management platform for students, supervisors, and administrators. The system adopts a front-end and back-end separation architecture built with Spring Boot + Vue3, using a unified label space of "student–supervisor–topic" as the data foundation to associate student profiles, supervisor profiles, and topic requirements. In terms of business process, the platform simultaneously supports two entry modes: "select topic first then review" and "select supervisor first then assign topic", which converge into a unified application entity on the back-end to ensure process flexibility and management consistency. The system achieves three key capabilities: a tag generation mechanism based on rule mapping and large model extraction; a matching algorithm that fuses tag overlap, semantic vectors, and supervisor profile similarity, with a fallback strategy to tag-only scoring when the embedding service is unavailable; and a proposition collaboration mechanism that follows the pipeline of AI-generated candidate topics, similarity-based deduplication, and supervisor manual finalization. During engineering implementation, transactional boundaries and database constraints jointly ensure consistent state transitions, while an admin-facing monitoring and log viewing interface is provided for traceable operations and troubleshooting. The platform can reduce the communication and screening costs of topic selection, improve matching efficiency between students and topics as well as between students and supervisors, and provide a sustainable data pipeline for subsequent quality evaluation and topic bank improvement.

Keywords: Graduation Thesis Topic Management; Supervisor Management; Label-based Modeling; Recommendation; Collaborative Management System


1 绪论

毕业论文（设计）选题管理在系统层面通常表现为“发布—申请—审核—绑定”的流程，但在真实运行里，问题往往不止在流程本身。题目信息散落在微信群、邮件、表格与口头沟通中，学生能看到题目，却很难判断门槛；导师在有限时间内反复命题，题目描述容易宽泛；过程记录缺少结构化沉淀，选题质量与指导效果难以在多届之间形成可比较的数据。

本文将选题管理表述为“学生—导师—选题”三方协同匹配与过程治理问题：用统一标签空间把零散描述变成可度量特征；把匹配、审核、协作、评价串成可追溯的数据链路，使系统既能记录结果，也能为解释与改进提供依据。

1.1 研究背景与问题提出

教育部等部门近年来的政策文件强调需推进教育管理与服务数字化转型，毕业论文（设计）管理被视为提升人才培养质量的重要环节之一[1-2]。但在高校日常运行中，选题环节常呈现“系统在、数据却不在”的状态：系统可以承载发布与归档，却难以覆盖选题沟通、名额实时同步与匹配判断，导致学生选择成本高，导师筛选压力大，管理员难以复盘。

选题内容本身也存在结构性压力。导师需要在教学与科研之外指导多名学生，题目打磨空间有限，容易出现“老题翻新”；学生在信息不充分的前提下做出判断，写作阶段再发现数据与工具基础不足，从而引发变更与返工。

这些现象说明：关键不只是“有没有平台”，而在于平台的信息模型能否支撑比较与决策。本文所要回答的是：能否构建统一的可计算特征空间，让选题从“凭感觉”走向“有依据、可复盘”？

1.2 国内外研究现状

相关研究主要沿三条路径展开：流程电子化与工作流治理[11]、推荐系统与信息检索的内容理解方法[3-6,22]、大语言模型与句向量带来的生成与语义检索方法[7-10,13]。综合来看，现有工作常在“算法精度”与“制度落地”之间割裂：偏算法的研究弱化真实约束，偏流程的系统缺少内容理解与可解释匹配。本文的切入点是把“算法”与“制度”一起纳入设计：统一标签空间作为数据底座，AI 能力以可控、可降级方式嵌入真实流程，使系统可运行、可治理、可演进。

1.3 研究目标与意义

本文的目标不是复现常见选题流程系统，而是把选题环节的零散问题串成可执行的工程链路：

- 在信息建模上，以统一标签空间表达学生能力与兴趣、导师研究方向与资源、选题能力要求与数据需求；
- 在匹配算法上，以可解释为优先，标签重合度作为基础得分，语义向量作为补充视图；向量服务不可用时回退到标签侧得分，保证主流程连续；
- 在流程上，把匹配结果嵌入发布、申请、审核与协作，并把评分与反馈沉淀为质量指标，为题库改进提供数据依据。


2 需求分析

2.1 角色与职责边界

- 学生：维护个人画像与标签，浏览选题/导师信息，提交申请，参与协作，上传论文与反馈。
- 导师：维护个人画像与标签，创建选题并提交审核，处理学生申请，指导协作环节，录入评价。
- 管理员：维护全局制度（开放时间/开关），审核选题，查看负荷与统计，监控系统运行与日志。

2.2 功能需求

系统功能按业务链路展开：

- 选题管理：导师创建/编辑选题，提交审核；管理员审核通过后开放。
- 双入口申请：学生可按选题申请，也可按导师发起意向申请；后端统一以申请关系作为锚点。
- 匹配与推荐：学生端列表展示匹配分数用于排序提示；导师端按匹配分数辅助处理申请。
- 变更与返工：学生可提交换题/换导师申请，过程留痕。
- 协作与材料：按环节上传文档，导师审核，通过/退回可追踪。
- 评价闭环：导师评分与评语、学生满意度与文字反馈；按选题聚合形成质量指标。
- 管理端监控：查看运行状态与日志，辅助排障。

2.3 非功能需求

- 可用性：embedding 或 AI 服务不可用时，不阻断核心选题与申请流程。
- 一致性：申请提交与名额计数、申请审核与占用约束保持一致。
- 可追溯：关键流程的状态变更与异常可定位。
- 安全性：按角色分界访问权限，跨角色资源访问在服务层再校验。
- 可维护：AI/embedding 能力通过配置开关与抽象层替换，避免业务模块散落调用。

2.4 关键数据对象

系统以 MySQL 存储业务数据。关键实体包括 `user`、`student_profile`、`teacher_profile`、`user_tag`、`topic`、`topic_application`、`thesis`、`thesis_evaluation`、`topic_metrics`、`change_request`、`mentor_application` 等。核心锚点是 `topic_application`：绑定学生、导师与选题，并作为协作与评价的数据关联入口。


3 系统框架设计

3.1 总体架构与技术选型

系统采用前后端分离架构：前端 Vue3 + TypeScript + Element Plus + Pinia 负责三端界面与状态管理；后端 Spring Boot + MyBatis-Plus 提供 REST API、封装事务与数据访问；数据库使用 MySQL 存储业务数据；AI 标签抽取与 AI 选题生成使用统一的大模型接口；语义向量通过本地 embedding 服务提供，后端以可降级策略调用。

其中，AI 与 embedding 的关键配置如下（摘录自 `backend/src/main/resources/application.yml`）：

```yaml
ai:
  tag:
    enabled: true
    api-url: https://gcli.ggchan.dev/v1/chat/completions
    api-key: ${AI_TAG_API_KEY}
    model: gemini-2.5-pro
    timeout-ms: 300000

embedding:
  enabled: true
  base-url: http://localhost:8000
```

3.2 数据库设计与一致性约束

为避免“一题多占”，系统在 `topic_application` 表中设置唯一性约束：同一选题最多允许 1 条 `APPROVED`。业务层在审批前也会先查询是否已有通过记录，两者叠加形成双保险。

3.3 模块划分与关键接口

系统按职责划分模块：选题模块（创建、审核、开放、去重）、申请模块（提交、处理、通知）、匹配模块（分数计算与降级）、协作模块（环节上传与审核）、评价模块（双向评价与指标聚合）、管理模块（制度设置、监控、日志）。

3.4 可降级策略

embedding 与 AI 均以“开关 + 失败返回空”的方式集成：当服务不可用时，业务流程仍可基于标签侧得分与规则逻辑继续运行。降级不是异常补丁，而是设计的一部分。

4.1 AI 标签抽取：可控调用与规则兜底

标签抽取封装在 `AiTagService`：未启用或调用失败时返回空列表，由上层回退到规则逻辑；同时引入短 TTL 内存缓存，减少重复点击造成的网关压力。

核心代码段（节选自 `backend/src/main/java/com/example/graduation/service/AiTagService.java`）：

```java
public List<String> extractTagNames(String text) {
    // 输入为空：直接返回空，避免无意义调用
    if (text == null || text.trim().isEmpty()) {
        return Collections.emptyList();
    }
    // 总开关关闭：让上层走规则兜底
    if (!properties.isEnabled()) {
        return Collections.emptyList();
    }

    String prompt = String.format(PROMPT_TEMPLATE, text.trim());

    // 同一 prompt 的短时重复点击：直接复用缓存结果，降低延迟与请求量
    List<String> cached = getCached(model, prompt);
    if (cached != null) {
        return cached;
    }

    try {
        // 调用大模型（此处省略具体 request/response 组装细节）
        ResponseEntity<Map<String, Object>> response = restTemplate.exchange(...);
        List<String> tags = parseTagNames(content.toString());

        // 写缓存：减少同一文本的重复推理
        if (!tags.isEmpty()) {
            putCache(model, prompt, tags);
        }
        return tags;
    } catch (Exception e) {
        // 失败不阻断主流程：上层会回退到规则抽取/手工编辑
        log.warn("AI 标签抽取请求失败，将使用规则抽取: {}", e.getMessage());
        return Collections.emptyList();
    }
}
```

4.2 AI 选题生成与命题协同：五维输出 + 去重过滤 + 导师定稿

AI 选题生成输出为“五维结构”，生成后不直接入库，而是先做去重检测，过滤后返回候选集供导师修改与导入。该流程的目的很清楚：让 AI 扩展思路，但不替代导师定稿。

接口层过滤逻辑（节选自 `backend/src/main/java/com/example/graduation/controller/TopicController.java`）：

```java
List<AiTopicService.CandidateTopic> candidates =
        aiTopicService.generateTopicsForTeacher(teacherId, requestDto);

List<AiGeneratedTopicResponse> responses = candidates.stream().map(c -> {
    // 每条候选题都与历史题库做去重对比
    TopicService.DuplicateCheckResult check =
            topicService.checkDuplicate(null, c.getTitle(), c.getDescription());

    AiGeneratedTopicResponse dto = new AiGeneratedTopicResponse();
    dto.setTitle(c.getTitle());
    dto.setDescription(c.getDescription());

    dto.setMaxSimilarity(check.getMaxSimilarity());
    dto.setSimilarTopicTitle(check.getSimilarTopicTitle());
    dto.setPassed(check.isPassed());
    return dto;
})
// 只保留通过阈值的候选题
.filter(AiGeneratedTopicResponse::isPassed)
// 去重过滤后按期望数量截断返回
.limit(desired)
.collect(Collectors.toList());
```

4.3 去重检测：Jaccard + 余弦融合与阈值判定

去重检测实现为轻量级算法：对标题+描述做简化分词，计算 Jaccard 与余弦相似度后融合，阈值默认 0.7。这里不追求复杂模型，目的是提供一条可解释、可复现的拦截线。

核心代码段（节选自 `backend/src/main/java/com/example/graduation/algo/DuplicateCheckAlgorithm.java`）：

```java
// 相似度阈值：>= 0.7 视为“重复风险高”，提交审核或 AI 候选返回时会被拦截/过滤
private static final double SIMILARITY_THRESHOLD = 0.7;

public double calculateSimilarity(String text1, String text2) {
    // tokenize() 为简化分词：工程上可运行，后续可替换为更精细的中文分词
    Set<String> words1 = tokenize(text1);
    Set<String> words2 = tokenize(text2);

    double jaccard = calculateJaccard(words1, words2);
    double cosine = calculateCosine(words1, words2);

    // 融合权重：余弦更关注整体“共同词项密度”，因此权重更高
    return 0.4 * jaccard + 0.6 * cosine;
}

public boolean isPassed(double similarity) {
    return similarity < SIMILARITY_THRESHOLD;
}
```

4.4 可降级的语义向量调用：失败回退与缓存

`EmbeddingServiceImpl` 以 HTTP 调用本地 `/embed`。当 `embedding.enabled=false` 或调用失败时，返回 null；上层匹配服务会回退到标签侧得分，保证列表与申请仍可继续。

核心代码段（节选自 `backend/src/main/java/com/example/graduation/service/impl/EmbeddingServiceImpl.java`）：

```java
public float[] embedText(String text) {
    // 语义向量服务总开关：关闭即不参与计算（用于降级或资源不足场景）
    if (!enabled) {
        return null;
    }
    if (text == null || text.isBlank()) {
        return null;
    }

    // 轻量归一化：提升缓存命中率
    String normalized = normalizeText(text);

    // 缓存：避免同一文本被反复向量化
    float[] cached = cache.get(normalized);
    if (cached != null) {
        return cached;
    }

    try {
        Map<String, Object> respBody = restTemplate.postForObject(url, entity, Map.class);
        // ... 解析响应中的 vector 数组为 float[] ...
        cache.put(normalized, result);
        return result;
    } catch (Exception e) {
        // 失败只影响“语义视图”，不阻断主业务：由上层回退到标签匹配
        log.warn("调用 Embedding 服务失败，将回退到标签匹配: {}", e.getMessage());
        return null;
    }
}
```

4.5 匹配算法：标签重合度 + 语义相似 + 导师画像相似度融合

匹配服务封装在 `MatchServiceImpl`，用于学生端列表展示与申请提交时的分数计算。融合逻辑把各子分数都约束在 0–1，再映射到 0.30–1.00 的展示区间。

核心融合代码段（节选自 `backend/src/main/java/com/example/graduation/service/impl/MatchServiceImpl.java`）：

```java
BigDecimal overlap = calculateOverlapScore(studentTags, text);          // 标签重合度（可解释）
BigDecimal semantic = calculateSemanticSimilarity(topic, studentTags);  // 语义相似（可降级）
BigDecimal teacherSimilarity = calculateTeacherStudentSimilarity(topic, studentTags); // 导师画像相似

// alpha：标签分数占比，强调可解释性；语义作为补充视图
BigDecimal alpha = new BigDecimal("0.6");
BigDecimal topicSide = overlap.multiply(alpha)
        .add(semantic.multiply(BigDecimal.ONE.subtract(alpha)));

// 与导师画像相似度再做一次等权融合，避免“只看题不看人”
BigDecimal fused = topicSide.multiply(new BigDecimal("0.5"))
        .add(teacherSimilarity.multiply(new BigDecimal("0.5")));

// 映射到 [0.30, 1.00]：避免极端低分对学生造成不必要的心理压力
BigDecimal score = new BigDecimal("0.30").add(fused.multiply(new BigDecimal("0.70")));
return score.setScale(2, RoundingMode.HALF_UP);
```

Abstract

The Ministry of Education has proposed policies to promote the full-process digital transformation of educational administration management, with graduation thesis (design) topic selection management being an important link to improve the quality of talent cultivation. Addressing the common problems of "information dispersion, coarse-grained matching, and difficult process tracking" in university graduation thesis topic selection, this paper designs and implements a collaborative topic selection management platform for students, supervisors, and administrators. The system adopts a front-end and back-end separation architecture built with Spring Boot + Vue3, using a unified label space of "student–supervisor–topic" as the data foundation to associate student profiles, supervisor profiles, and topic requirements. In terms of business process, the platform simultaneously supports two entry modes: "select topic first then review" and "select supervisor first then assign topic", which converge into a unified application entity on the back-end to ensure process flexibility and management consistency. The system achieves three key capabilities: a tag generation mechanism based on rule mapping and large model extraction; a matching algorithm that fuses tag overlap, semantic vectors, and supervisor profile similarity, with a fallback strategy to tag-only scoring when the embedding service is unavailable; and a proposition collaboration mechanism that follows the pipeline of AI-generated candidate topics, similarity-based deduplication, and supervisor manual finalization. During engineering implementation, transactional boundaries and database constraints jointly ensure consistent state transitions, while an admin-facing monitoring and log viewing interface is provided for traceable operations and troubleshooting. The platform can reduce the communication and screening costs of topic selection, improve matching efficiency between students and topics as well as between students and supervisors, and provide a sustainable data pipeline for subsequent quality evaluation and topic bank improvement.

引言

高校毕业论文（设计）选题管理看似只是“发布—申请—审核”的流程问题，但在真实运行中，它往往牵扯出更深层的三类矛盾：其一，信息以碎片化形式散落在微信群、邮件、表格与口头沟通之中，导致“看得到题目，却看不清要求”；其二，选择机制偏粗粒度，学生与题目、学生与导师的匹配更多依赖经验与直觉，难以解释也难以复盘；其三，过程数据缺少结构化沉淀，使得选题质量与指导效果很难形成长期积累与持续改进。

在此背景下，本文将毕业论文选题管理重新表述为“学生—导师—选题”三方协同匹配与过程治理问题：一方面，通过统一标签空间把原本零散的自然语言描述转化为可度量、可比较的特征；另一方面，围绕真实教务流程把匹配、审核、协作、评价等环节串成可追溯的数据链路，使系统不仅能“记录发生了什么”，也能为“为什么会这样、怎样可以更好”提供更可解释的支撑。


1 研究背景与意义

1.1 研究背景与问题提出

教育部等部门近年来的政策文件强调需以学生发展为着眼点推进教育管理与服务数字化转型，毕业论文（设计）管理被明确当作提升人才培养质量的重要环节之一[1-2]。然而在高校的日常运行中，选题环节常呈现出一种“系统在、数据却不在”的尴尬：多数高校虽上线了选题发布、材料提交、审核归档等功能，但在选题沟通与筛选阶段，仍大量依赖线下或半线下方式完成，信息同步滞后、过程记录不完整，进而放大了“热门资源被提前占用、导师难以把握实时名额、学生难以准确判断题目门槛”等问题。

更棘手的是，选题本身还存在“题目陈旧、描述宽泛、要求不清”的结构性压力。导师在教学、科研与指导任务并行的情况下，往往难以每年从头打磨大量高质量选题，导致“老题翻新”在多届中反复出现；学生则容易在缺少信息的前提下做出偏乐观判断，直到开题或写作阶段才发现数据来源、工具基础与题目要求不匹配，从而引发变更、返工甚至质量下滑。

这些现象说明：选题管理的关键并不只是“有没有平台”，而在于平台的信息模型是否足以支撑比较与决策。若系统仅记录“谁选了什么、状态如何”，则无法支持学生、导师与管理者在过程中的解释性判断与持续优化。本文所关注的核心问题因此落在：能否构建一个统一的可计算特征空间，让选题决策从“凭感觉”走向“有依据、可复盘”？

1.2 国内外研究现状

相关研究大体可分为三条路径：其一是流程电子化与工作流治理，将线下表格与签字搬到线上，并通过流程节点强化责任边界与过程可追踪性[11]；其二是推荐系统与信息检索方向的内容理解方法，通过关键词、主题模型、向量空间模型等手段衡量学生与项目之间的匹配程度[3-6,22]；其三是随着大语言模型与句向量技术兴起的生成与语义检索方法，用模型辅助方向推荐、内容归纳与相似度检索[7-10,13]。

综合来看，现有工作往往在“算法精度”与“制度落地”之间存在割裂：偏算法的研究容易弱化真实约束与流程边界，偏流程的系统则缺少内容理解与可解释匹配能力。本文的切入点在于：从设计之初就把学生、导师与选题统一到同一标签空间，让匹配算法与管理流程共用同一套数据底座，同时把 AI 能力以可控、可降级的方式嵌入进去，使系统在真实场景中可运行、可治理、可演进。

1.3 研究目标与意义

本文的目标并非复现一个常见的“选题流程系统”，而是尝试把选题环节中长期存在的零散问题，串成一条可执行的工程链路。具体而言：

- 在信息建模上，引入统一标签空间，将学生能力与兴趣、导师研究方向与资源、选题能力要求与数据需求用同一语言表达，形成可比较特征；
- 在算法上，以“可解释”为优先，采用标签重合度为基础，并引入向量相似度作为补充视图；在向量服务波动时，系统能回退到标签侧得分，避免把主业务绑死在不稳定服务上；
- 在流程上，把匹配结果嵌入到发布、申请、审核与协作中，并将评分与反馈沉淀为质量指标，为后续题库改进与制度优化提供数据支撑。

这一研究路径强调“门槛不高，但能长期生长”：在不追求堆砌复杂模型的前提下，通过数据结构化、规则约束与可演进架构，为高校教务治理提供可落地的智能化改造范式。


2 研究设计与技术路线

2.1 设计目标与总体思路

围绕高校毕业论文选题管理中“信息分散、匹配粗放、过程难追踪”等短板，系统总体思路可概括为三段连续链路：第一，构建学生、导师与选题的标签化画像，使其在统一特征空间中被刻画；第二，在该空间上设计匹配与生成方法，为“学生—选题—导师”三方提供可解释的匹配判断，同时将 AI 能力限制在“人机协同、人工定稿”的可控框架内；第三，把上述方法嵌入真实教务流程，使每次选题、协作与评价都沉淀为可追溯数据，形成闭环评价链路[24-25]。

2.2 角色职责与核心业务边界

系统围绕三类角色展开：学生负责完善画像、发起申请、参与协作并提交论文；导师负责命题与维护、审核与分配、指导与评分；管理员负责制度设置、选题审核与全局监控。为了避免“系统只记录结果”的局限，系统将关键业务动作统一锚定在“申请记录”和“协作环节”上，使得题目、导师、学生的关系不仅可被确定，也可被追踪与复盘。

2.3 学生侧标签画像与推荐逻辑设计

标签画像的目的并不是用一个黑盒子替学生排序，而是把“更适合的题目”以一种不打断自主选择的方式推到前面。系统将学生标签与选题文本映射到同一向量空间：标签命中用于解释“为什么相似”，向量相似用于缓解“同义改写导致命中下降”的问题；当向量服务不可用时，系统以标签侧得分兜底，保证申请与浏览功能不受影响。

2.4 导师 AI 选题生成与去重机制

导师命题是选题质量的源头，也是结构性压力最集中的环节。系统将大语言模型定位为“命题助手”：它负责在约束下生成结构化草稿，而题目是否进入题库、表述如何打磨，最终仍由导师人工定稿。为避免“生成一堆重复题”，系统在生成后对候选题执行去重检测，超过阈值的候选题将被过滤，减轻导师筛选负担。

2.5 闭环评价与数据链路设计

系统在“论文提交—导师评分—学生反馈”处引入双向评价：导师侧评分与评语用于形成质量指标，学生侧满意度与文字反馈用于补足过程体验。评价数据按选题聚合，沉淀到质量指标表中，为题库迭代与管理决策提供依据。


3 系统实现

本章按“一个选题季”的时间线说明系统功能落地方式：选题发布与审核、双模式申请合流、协作与提交、评价闭环与可视化。与仅做功能罗列不同，本文更关注“流程为何这样设计”以及“数据如何沉淀”。

3.1 选题发布与审核

导师在“选题管理”模块创建或编辑选题，完成标题、描述、人数限制与标签；提交审核时系统先做去重检测，再进入管理员审核队列。管理员审核通过后选题开放，进入学生端列表。

3.2 双模式申请流程：先选题与先选导师

系统提供“选题中心”和“导师列表”两种入口：前者强调题目本身，后者强调导师画像。两条路径在后端统一落到“申请记录”这一实体上，使审批、绑定与协作的锚点一致，避免重复建模。

3.3 选题匹配与排序建议

匹配分数在系统里以“排序参考”的角色出现：学生端用于题目列表的推荐提示，导师端用于申请列表的优先级建议。分数本身可拆解为标签命中与语义相似等部分，以降低“黑箱排序”带来的抵触。

3.4 选题变更管理

系统提供受控的变更申请（更换选题/更换导师）入口，并将变更请求与当前生效的申请记录绑定。这样设计既避免“完全放任变更”破坏稳定性，也防止变更过程回到线下导致记录缺失。

3.5 协作与过程材料沉淀

协作中心将文档上传与消息沟通集中在同一页面，所有消息与文档均与申请记录绑定持久化，便于导师集中审阅，也便于后续分析“高风险环节”与“返工热点”。

3.6 评价与反馈：从终点到起点

成绩录入与学生反馈并不只是流程终点，它们被系统视作下一轮改进的起点：评价数据沉淀后按选题聚合，形成长期指标，帮助导师回顾题目设计与指导成效，帮助管理者识别结构性风险。

系统采用前后端分离架构：前端 Vue3 + TypeScript + Element Plus + Pinia 负责三端界面与状态管理；后端 Spring Boot + MyBatis-Plus 负责业务 API、数据访问与事务；数据库 MySQL 存储业务数据；AI 标签抽取与 AI 选题生成通过统一的大模型接口实现；语义向量通过本地 embedding 服务提供，后端以可降级策略调用。

其中，embedding 与 AI 的关键配置项如下（摘录自 `backend/src/main/resources/application.yml`）：

4.2 AI 标签抽取：可控调用与规则兜底

标签抽取对学生画像与导师画像都很关键，但它不应成为“必须成功”的硬依赖。系统将大模型抽取封装在 `AiTagService` 中：若未启用或调用失败，直接返回空列表，由调用方回退到规则逻辑。服务内部还引入了短 TTL 的内存缓存，以降低重复点击带来的网关压力与体感延迟。

核心代码段如下（节选自 `backend/src/main/java/com/example/graduation/service/AiTagService.java`）：

```java
public List<String> extractTagNames(String text) {
    if (text == null || text.trim().isEmpty()) {
        return Collections.emptyList();
    }
    if (!properties.isEnabled()) {
        return Collections.emptyList();
    }

    String prompt = String.format(PROMPT_TEMPLATE, text.trim());
    // cache hit
    List<String> cached = getCached(model, prompt);
    if (cached != null) {
        return cached;
    }

    try {
        ResponseEntity<Map<String, Object>> response = restTemplate.exchange(...);
        List<String> tags = parseTagNames(content.toString());
        if (!tags.isEmpty()) {
            putCache(model, prompt, tags);
        }
        return tags;
    } catch (Exception e) {
        log.warn("AI 标签抽取请求失败，将使用规则抽取: {}", e.getMessage());
        return Collections.emptyList();
    }
}
```

这里的关键点不在于“抽取一定准确”，而在于两件事：第一，失败时不影响主流程；第二，返回结果经过清洗过滤，避免把“错误信息、请求词、日志词”等噪声写入标签库，降低后续匹配算法被污染的概率。

4.3 AI 选题生成：五维模板输出 + 去重过滤 + 导师定稿

导师侧 AI 选题生成遵循“先结构化约束，再生成草稿”的思路。`AiTopicService` 中定义了统一的 Markdown 模板，要求每条候选题必须包含“研究背景、主要内容、能力要求、数据/资源需求、创新点”五个维度，以便后续用于审核与统计。

生成后系统并不会直接入库，而是先在接口层对每条候选题执行去重检测，仅返回通过阈值的候选集给前端。对应逻辑在 `TopicController.generateAiTopics()` 中体现：

```java
List<AiTopicService.CandidateTopic> candidates = aiTopicService.generateTopicsForTeacher(teacherId, requestDto);
List<AiGeneratedTopicResponse> responses = candidates.stream().map(c -> {
    TopicService.DuplicateCheckResult check =
        topicService.checkDuplicate(null, c.getTitle(), c.getDescription());
    AiGeneratedTopicResponse dto = new AiGeneratedTopicResponse();
    dto.setTitle(c.getTitle());
    dto.setDescription(c.getDescription());
    dto.setMaxSimilarity(check.getMaxSimilarity());
    dto.setSimilarTopicTitle(check.getSimilarTopicTitle());
    dto.setPassed(check.isPassed());
    return dto;
}).filter(AiGeneratedTopicResponse::isPassed)
  .limit(desired)
  .collect(Collectors.toList());
```

这种“先多生成、再过滤、再限量返回”的策略，能够在阈值过滤存在损耗时，仍尽量保证返回数量接近导师期望值；同时把“相似度最大值、相似题标题”带回前端，便于导师直观看到被过滤的原因。

4.4 去重算法：Jaccard + 余弦融合与阈值判定

去重检测并不是为了追求复杂度，而是为了在工程上提供一个稳健的“第一道门”。系统实现了一个轻量级文本去重算法 `DuplicateCheckAlgorithm`：对标题+描述文本进行简化分词后，计算 Jaccard 相似度与余弦相似度，并按权重线性融合，阈值默认为 0.7。

核心代码段如下（节选自 `backend/src/main/java/com/example/graduation/algo/DuplicateCheckAlgorithm.java`）：

```java
private static final double SIMILARITY_THRESHOLD = 0.7;

public double calculateSimilarity(String text1, String text2) {
    Set<String> words1 = tokenize(text1);
    Set<String> words2 = tokenize(text2);
    double jaccard = calculateJaccard(words1, words2);
    double cosine = calculateCosine(words1, words2);
    return 0.4 * jaccard + 0.6 * cosine;
}

public boolean isPassed(double similarity) {
    return similarity < SIMILARITY_THRESHOLD;
}
```

在 `TopicService.submitForReview()` 中，去重检测作为“提交审核”的前置条件执行：相似度超过阈值则直接提示修改后再提交，从而把问题尽量拦截在“公开之前”，避免题库重复在学生端扩散。

4.5 可降级的语义向量调用：失败回退与缓存

语义向量服务通过 `EmbeddingServiceImpl` 以 HTTP 方式调用本地 `/embed` 接口。该实现遵循两条工程原则：第一，可关闭（`embedding.enabled=false` 时直接返回 null）；第二，可失败（异常时返回 null，并记录 warn 日志），由上层逻辑回退到标签侧计算。

核心代码段如下（节选自 `backend/src/main/java/com/example/graduation/service/impl/EmbeddingServiceImpl.java`）：

```java

    if (cached != null) {
```

4.6 匹配算法：标签重合度 + 语义相似 + 导师画像相似度融合

匹配算法的目标关键词是：可解释、可控、可演进。系统将其封装在 `MatchServiceImpl`，并在 `TopicController.open-with-score` 与 `ApplicationController.submitApplication` 这两条关键路径中被调用。

匹配分数由三部分构成：

- 学生标签 ↔ 选题文本：标签权重重合度（可解释）；
- 学生标签 ↔ 选题文本：语义相似度（embedding 可用时参与，不可用时回退）；
- 学生标签 ↔ 导师画像：导师研究方向与导师标签的语义相似度（embedding 不可用时返回中性值 0.5）。

核心融合逻辑如下（节选自 `backend/src/main/java/com/example/graduation/service/impl/MatchServiceImpl.java`）：

```java
BigDecimal overlap = calculateOverlapScore(studentTags, text);
BigDecimal semantic = calculateSemanticSimilarity(topic, studentTags);
BigDecimal teacherSimilarity = calculateTeacherStudentSimilarity(topic, studentTags);

BigDecimal alpha = new BigDecimal("0.6");
BigDecimal topicSide = overlap.multiply(alpha)
    .add(semantic.multiply(BigDecimal.ONE.subtract(alpha)));

BigDecimal fused = topicSide.multiply(new BigDecimal("0.5"))
    .add(teacherSimilarity.multiply(new BigDecimal("0.5")));

BigDecimal score = new BigDecimal("0.30").add(fused.multiply(new BigDecimal("0.70")));
return score.setScale(2, RoundingMode.HALF_UP);
```

这里的“0.30–1.00 映射”并非为了做数学花活，而是出于产品与心理层面的取舍：避免给学生造成极端低分的挫败感，同时让分数分布更平滑、更易于在列表中呈现“差异但不过分绝对”的排序参考。

4.7 事务化状态转变：申请提交、审核与唯一性约束

系统把“申请提交”视为一次关键的状态写入，因此在 `ApplicationService.submitApplication()` 中采用事务包裹：插入 `topic_application` 记录与更新 `topic.current_applicants` 在同一事务内完成，避免出现“申请写入了但人数未更新”或相反的分离状态。

关键代码段如下（节选自 `backend/src/main/java/com/example/graduation/service/ApplicationService.java`）：


6 结论与展望

本文围绕高校毕业论文选题管理的现实痛点，提出并实现了一套以“统一标签空间”为纽带的协同选题管理平台。在方法上，系统以标签重合度为基础，融合语义向量与导师画像相似度形成可解释匹配，并通过 embedding 降级策略保证主业务连续；在工程上，系统以事务边界与数据库约束保障关键状态写入的一致性，并用管理端监控与日志接口缩短排障链路；在治理上，通过评价闭环把选题质量与指导效果沉淀为长期指标，为题库维护与制度优化提供数据支撑。

未来工作可沿两条线扩展：一是进一步完善过程环节（开题、中期、答辩、成果转化等）的统一建模，使全过程可追踪；二是增强治理与审计能力，例如引入独立的操作审计表，完善关键写操作的结构化日志化记录，并在更大规模运行数据基础上，对匹配权重与阈值策略做持续校准与优化。
