# 标签算法说明与 AI 赋能

本文档说明当前项目中**标签的生成与使用方式**、**为何在实际使用中不够严谨**，以及**如何通过 AI 增强标签步骤**。

---

## 一、标签在项目中的角色

| 类型 | 存储表 | 来源 | 用途 |
|------|--------|------|------|
| **用户标签**（导师） | `user_tag` | 由「研究方向」自动生成 | 导师画像、AI 选题生成时用 |
| **用户标签**（学生） | `user_tag` | 由「专业」+「兴趣描述」自动生成 | 学生画像、AI 选题生成时用 |
| **选题标签** | `topic_tag` | 导师创建/编辑选题时**手动输入** | 选题分类、展示；设计上可用于学生-选题匹配 |
| **匹配度** | `topic_application.match_score` | 目前**写死为 0.5** | 申请列表“按匹配度排序”实际未用标签计算 |

---

## 二、当前标签算法运作情况

### 2.1 用户标签的生成（`TagService`）

- **入口**：导师完善资料时传 `researchDirection` → `generateTeacherTags()`；学生完善资料时传 `interestDesc`、`major` → `generateStudentTags()`。
- **核心逻辑**：`extractTags(String text, BigDecimal defaultWeight)`  
  - 仅用 **`String.split("[,，、\\s]+")`** 按逗号、顿号、空格分词。  
  - 对每一段 `trim()` 后，只保留 **长度 2–6** 的字符串作为标签名。  
  - 导师：整段研究方向共用权重 0.9；学生：专业 0.8、兴趣 0.9，再去重合并、同名校准取最大权重。

代码位置：`backend/.../service/TagService.java`（约 84–99 行）。

### 2.2 选题标签的生成（`TopicService`）

- **入口**：创建/更新选题时，前端传 `List<String> tags`，由导师**手动输入**（如输入「机器学习」「推荐系统」）。
- **逻辑**：`saveTopicTags()` 只做长度 2–6 校验，通过则写入 `topic_tag`，权重固定 0.5。  
  - 没有从「标题 + 描述」中自动抽标签，也没有和用户标签做归一化。

### 2.3 标签与匹配度的关系

- 数据库中有 `topic_application.match_score`，设计上可用于“学生-选题匹配度”。
- 当前实现中：**匹配度未基于标签计算**，提交申请时在 `ApplicationController` 中写死为 `0.5`，因此「按匹配度排序」并没有体现标签或算法效果。

## 三、为何“实际使用时并没有很严谨的体现”

1. **无真正中文分词**  
   仅按标点/空格切分，例如「机器学习与推荐系统」若用户不写逗号，会变成一整段被长度 2–6 过滤掉或整段当一个“词”，标签质量依赖用户是否自己用逗号、顿号分隔。

2. **无停用词与归一化**  
   「的」「与」「和」「研究」「应用」等未过滤；同义或近义（如「深度学习」「深度神经网络」）不会合并，导致标签分散、匹配时难以利用。

3. **选题标签完全依赖人工**  
   选题只有手动填的标签，没有从标题、描述中自动抽取，容易漏标或与用户标签用词不一致，后续难以做基于标签的匹配或推荐。

4. **匹配度未用标签**  
   学生-选题匹配度固定 0.5，没有利用 `user_tag` 与 `topic_tag` 做相似度或权重加权，所以标签在业务流程中“存在但未真正参与决策”。

5. **权重与使用脱节**  
   用户标签有权重（0.8/0.9），但当前没有接口按“用户标签 × 选题标签”计算匹配分，权重只在存储层面存在。

## 四、AI 赋能在标签这一步可以做什么

在**不改变现有表结构**的前提下，可以在以下环节用 AI 增强：

| 环节 | 做法 | 效果 |
|------|------|------|
| **导师/学生用户标签** | 用大模型从「研究方向」「兴趣描述」「专业」中**自动抽取 2–6 个关键词**，替代或补充现有 split 逻辑 | 减少对逗号分隔的依赖，提升标签一致性和覆盖面 |
| **选题标签** | 导师填写标题+描述后，用 AI **推荐候选标签**，导师可勾选或修改后再提交 | 统一用词、减少漏标，便于后续做基于标签的匹配 |
| **匹配度（可选）** | 在提交申请时，用「学生 user_tag」与「选题 topic_tag」计算相似度（或再结合 AI 语义相似度）写回 `match_score` | 让“按匹配度排序”真正反映标签/语义匹配 |

本项目已提供**用户标签的 AI 抽取**（见下文实现说明）：从一段自由文本中调用大模型 API 抽取标签，并接入现有 `TagService`，在“生成导师/学生标签”时优先使用 AI 结果，失败或未配置时回退到原有 split 逻辑。选题标签的 AI 推荐与匹配度算法可按同一思路扩展。

---

## 五、实现说明（AI 用户标签抽取）

- **配置**：在 `application.yml` 中增加 `ai.tag` 相关配置（是否启用、API 地址、Key 等），未配置或未启用时行为与改造前一致。
- **调用方式**：通过 HTTP 调用兼容 OpenAI 的聊天接口（或其它大模型），请求中传入“从以下文本中抽取 2–6 个关键词作为标签，仅返回逗号分隔的标签”的 prompt，解析返回内容得到 `List<String>` 再交给现有权重与去重逻辑。
- **集成点**：`TagService` 在 `extractTags()` 前若启用 AI，则先调用 AI 抽取，得到非空结果则以此为主（并可选择与 split 结果合并去重）；否则仍用原 split 逻辑。
- **依赖**：仅使用 Spring 自带的 HTTP 客户端（如 `RestTemplate`），无需额外 NLP 库；大模型 API 需在部署环境中配置 Key 与地址。

### 已实现内容

- **配置**：`application.yml` 中 `ai.tag` 节点  
  - `enabled`：是否启用（默认 `false`）  
  - `api-url`：兼容 OpenAI 的聊天接口 URL  
  - `api-key`：API 密钥（建议用环境变量 `AI_TAG_API_KEY` 覆盖）  
  - `model`：模型名（如 `gpt-3.5-turbo`）  
  - `timeout-ms`：请求超时毫秒数  

- **类**：  
  - `AiTagProperties`：绑定上述配置。  
  - `AiTagService`：根据 prompt 调用大模型，解析返回内容为「逗号分隔的标签」并过滤长度 2–6。  
  - `TagService.extractTags()`：若启用且 AI 返回非空，则用 AI 结果生成 `UserTag` 列表；否则回退到原有 `split` 逻辑。

- **使用**：将 `ai.tag.enabled` 设为 `true` 并配置有效的 `api-url`、`api-key` 后，导师/学生完善资料时的「研究方向」「兴趣描述」「专业」会优先经 AI 抽取标签再落库；未配置或调用失败时行为与改造前一致。
