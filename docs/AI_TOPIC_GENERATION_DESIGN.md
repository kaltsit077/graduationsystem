## 导师 AI 选题生成：设计理念与工作流草案

> 本文档用于记录当前关于“导师部分需求 + AI 补全 + 多轮生成”的整体设计思路，后续实现与论文撰写时以此为基准逐步细化。

### 1. 问题背景与改造动机

- **现状问题**
  - 现有 `AiTopicService` 的提示词以「导师标签 + 偏好说明」为主，输出格式为 `标题|背景与主要内容|关键词1,关键词2,关键词3`。
  - 实际生成效果偏“标题生成 + 一段模糊描述”，难以支撑论文中描述的“完整、可落地的候选选题结构”，也无法很好引导导师思考“能力要求、数据需求、创新点”等关键要素。
  - 前端界面的“偏好说明”字段缺乏明确填写指引，导师容易随手写几句，导致大模型生成内容的可控性和可用性都有限。

- **目标场景**
  - 导师只需要**锁定少量核心约束**（典型如：研究领域 / 核心方法 / 应用场景），其余细节交给 AI 在约束内补全。
  - AI 必须按照统一的结构化模板输出一个**“可直接入库的候选选题草案”**，而不是泛泛的灵感提示。
  - 导师可以通过**多轮生成 + 局部修改**逐步逼近理想题目，形成“人机协同命题”的闭环，而不是每次从零开始重写。

### 2. 目标效果与输出结构

- **统一输出模板（面向导师）**

  模型生成的每条候选选题都遵循以下 Markdown 结构（字段名与顺序固定）：

  ```markdown
  ### 选题名称
  [简洁明了的题目，包含研究方法 + 应用场景]

  - 研究背景：[结合给定的行业/业务背景，说明选题意义，约 100 字；若未提供背景说明，则根据导师标签与典型业务场景合理假设]
  - 主要内容：[拆解 3–4 个关键研究要点，用短句概括，如“构建 ×× 模型；设计 ×× 算法；搭建 ×× 原型系统”等]
  - 能力要求：[学生需要具备或愿意学习的主要算法、方法和工具，如“线性/整数规划”“Python + Gurobi”“时间序列预测”等]
  - 数据/资源需求：[完成该题至少需要的数据类型或资源条件，如“近 3 个月门店销售和库存数据”“可访问企业导师进行访谈”等；若未提供，则根据场景给出不过分理想化的假设]
  - 创新点：[与传统课程作业或常见论文选题相比的区别，可是方法上的组合创新、场景的精细化建模，或预测 + 决策 / 仿真 + 可视化等环节的有效结合，避免空泛表述]
  ```

- **字段可选输入，输出字段不可缺失**
  - 前端对应提供若干**选填输入框**（研究背景、主要内容要点、能力要求、数据/资源、创新点等），导师可以只填写其中一部分，甚至全部不填。
  - 无论导师填写情况如何，AI 输出端都必须完整包含上述五个字段，由 Prompt 中的补全规则控制“信息缺失时如何合理假设”，而不是简单留空或省略字段。

- **分层约束理念**
  - **强约束**：导师明确填写的技术 / 场景 / 能力要求等，模型必须在选题名称和对应字段中体现，不可偏离。
  - **中约束**：导师画像中的研究方向标签（如“运筹学”“数据挖掘”），模型需整体贴合该方向，但可在子领域内灵活变换。
  - **弱约束**：导师未填写的字段，由模型根据强/中约束自动补全，按照预设规则偏向“典型且可落地”的设定。

### 3. 预期工作流（导师视角）

1. 导师进入“AI 生成候选选题”界面。
2. 系统自动读取导师的标签画像（研究方向、指导风格等），并展示为“可参与生成的标签”，导师可勾选若干。
3. 导师填写或选择以下信息（全部为**非必填**，但界面文案会强调“信息越完整，生成结果越贴近实际需求”）：
   - 行业 / 业务背景说明（如：奶茶店库存与订货管理）
   - 希望覆盖的研究要点（如：需求预测 + 补货决策 + 仿真评估）
   - 对学生能力 / 工具的要求（如：整数规划、Python + Gurobi）
   - 可提供的数据 / 资源（如：3 个月销售与库存数据）
   - 期望的创新点（如：结合时序预测的动态补货策略）
   - 生成数量（如：5 条）
4. 系统将导师标签 + 以上填写内容组合为结构化提示词，调用大模型生成若干条**结构化候选选题**。
5. 生成结果以统一 Markdown 结构展示，导师可以：
   - 将满意的条目“一键导入选题列表”；
   - 对不满意的候选进行人工编辑（标题或各字段文本）；
   - 发起下一轮生成（后续可扩展为“基于上一轮结果优化生成”）。

    - **兼容与演进**

  - **编辑选题对话框（创建/编辑选题）**
    - 当选题描述为 AI 生成的五维 Markdown 时，自动识别并支持**分段编辑**：研究背景、主要内容、能力要求、数据/资源需求、创新点 五个输入框，便于导师逐项修改。
    - 提供「分段编辑」与「全文编辑」切换：全文编辑可查看/修改原始 Markdown；保存时分段模式会将五维内容重新组装为同一 Markdown 写回 `description`，无需改后端或数据库。

      - 强调“AI 只提供候选方案，最终由导师甄别与编辑后入库”，与后续去重与审核逻辑衔接。

### 5. 近期实施优先级（MVP）

为控制复杂度，近期实现聚焦以下“最小可用版本”：

- **必须实现**
  - 统一 Markdown 结构输出的 Prompt 设计与后端调用逻辑调整。
  - 前端新增多维度选题需求输入框（可选），并与后端新参数打通。
  - Markdown → `CandidateTopic` 的基础解析（至少解析出标题和完整正文），确保生成结果可以直接导入题库并在选题详情中展示。

- **可在论文展望中描述、后续逐步实现**
  - 基于上一轮结果的“优化生成”能力（轻量会话记忆）。
  - 字段级重新生成与局部编辑（增量生成接口）。
  - 生成历史与版本对比、与既有题库以及历史生成记录的去重增强。

### 6. 与整体系统设计的关系

- 本设计是对原有“导师 AI 选题生成与去重机制”的一次结构化升级：
  - **在表示层面**：从“短文本提示 + 模糊描述”升级为“面向选题全生命周期的结构化草案”，为后续质量评价与复用提供更好的信息基础。
  - **在交互层面**：从“单次一键生成”升级为“导师需求建模 + AI 补全 + 人机协同筛选”的过程，更符合高校教师的真实使用习惯。
  - **在算法层面**：结合标签画像、结构化需求与语义去重，进一步减少选题同质化风险，使 AI 生成部分真正成为可控的“生成助手”而非不可解释的“黑盒命题者”。

### 7. 算法实现与问题排查

#### 7.1 生成链路（算法流程）

1. **入口**  
   `TopicController.generateAiTopics` 接收 `AiGenerateTopicsRequest`（生成数量、参与标签、各维度需求说明等），调用 `AiTopicService.generateTopicsForTeacher(teacherId, requestDto)`。

2. **参数与需求组装**  
   - 从 `user_tag` 表读取该导师的标签；若请求中指定了 `tagNames`，则只保留这些标签，否则用全部标签。  
   - `buildDemandText(requestDto)` 将各选填字段（行业/业务背景、主要研究要点、能力/工具要求、数据/资源、创新点期望、其他说明）拼成一段「选题需求说明」文本；若全部为空则用 `preferenceHint`。

3. **是否走大模型**  
   - 若标签为空：直接返回空列表（不调用大模型，也不走本地模板）。  
   - 若 `ai.tag.enabled` 为 false，或 `apiUrl`/`apiKey` 未配置：不调用大模型，直接进入**本地模板回退**。  
   - 否则：用 `TOPIC_PROMPT_TEMPLATE` 拼出 prompt（导师标签、生成数量、选题需求说明），请求 `ai.tag.api-url` 的 chat completions 接口。

4. **大模型响应处理**  
   - 从响应中取 `choices[0].message.content`；若缺失或为空，记日志并返回空列表，上层进入回退。  
   - 对 `content` 做 `parseAiTopics(raw, count)`：  
     - 若被包在 markdown 代码块（\`\`\`）内，先去掉围栏再按行解析。  
     - 按行扫描，识别以 `### ` 开头的块：  
       - `### 选题名称` 时，下一行非空内容作为**真实题目**，再往后为描述行（含五个 `- 字段名：...`）；  
       - `### 实际题目`（其他内容）时，该行去掉 `### ` 即为题目，后续为描述行。  
     - 每个块得到 `title` + 整段描述（含五维 Markdown），构造 `CandidateTopic`；若未解析出任何一条，返回空列表。

5. **回退（本地模板）**  
   当大模型未启用、请求异常或解析结果为空时：  
   - 用当前标签随机组合成题目标题（如「基于 X 的 Y 系统研究」）。  
   - 描述固定为一句：「这是一个围绕「X」与「Y」展开的研究选题，具有一定的学术价值和实践意义。」+ 若有需求说明则追加「本题目偏向：…」。  
   - **不包含**五维结构，因此若界面上只看到这句话，说明本次走的是回退而非大模型结果。

6. **去重与返回**  
   - 对每个 `CandidateTopic` 调用 `TopicService.checkDuplicate`（与现有题库做相似度比对），仅保留 `passed=true` 的项。  
   - 返回给前端的 `AiGeneratedTopicResponse` 包含 `title`、`description`、`tags`、相似度等；导师点击「导入为我的选题」时，前端将 `title`、`description`、`tags` 原样传给 `POST /topics` 创建选题，故编辑页展示的即本次生成/回退写库的内容。

#### 7.2 为何界面上“只有这种程度”（仅标题 + 一句话描述）

- **直接原因**：当前请求实际走的是**本地模板回退**，没有使用大模型返回的五维 Markdown。  
- **可能原因**（按优先级排查）：  
  1. **大模型未启用或未配置**  
     - 检查 `application.yml` 中 `ai.tag.enabled` 是否为 `true`，且 `ai.tag.api-url`、`ai.tag.api-key` 已正确配置。  
     - `AiTagProperties.isEnabled()` 会在 `enabled && apiUrl 非空 && apiKey 非空` 时才为 true。  
  2. **请求大模型失败**  
     - 网络超时、4xx/5xx、鉴权失败等会抛异常，`generateWithAi` 捕获后返回空列表并打 `log.warn("AI 选题生成 请求异常，将使用本地模板生成: ...")`。  
     - 查看后端日志中的该条与异常栈，确认是超时、密钥错误还是上游返回错误 body。  
  3. **上游返回格式与解析不匹配**  
     - 若接口返回的 `choices[0].message.content` 不是“按行、以 `### ` 开头的 Markdown”，解析会得到 0 条。  
     - 例如：模型返回了前言（“以下是…”）、或把内容放在代码块内但格式与预期不符、或只输出了题目没有五维字段。  
     - 日志会打 `log.warn("AI 选题生成 解析失败: ... 前200字=[...]")`，可根据“前200字”调整 prompt 或解析的容错（如已支持去掉 \`\`\` 围栏、支持“### 选题名称”下一行为真实题目）。  
  4. **全部被去重过滤**  
     - 若大模型确实返回了多条且解析成功，但 `checkDuplicate` 全部判为与历史过于相似，则返回给前端的列表为空；此时不会显示“一句话描述”，而是无结果。因此“编辑页只有一句话”通常对应 1～3，不是 4。

#### 7.3 建议排查步骤

1. 看后端日志：是否有 `AI 选题生成 开始`、`AI 选题生成 成功`（大模型路径），还是 `回退到本地模板`、`解析失败`、`请求异常`。  
2. 确认配置：`ai.tag.enabled=true`，且同一配置下标签抽取等其它 AI 功能是否正常。  
3. 若为解析失败：根据日志中的“前200字”确认模型实际输出格式，必要时微调 prompt（强调“直接输出、不要前言”）或再放宽解析（如兼容 `## ` 或字段名微小差异）。  
4. 若为大模型路径成功：确认前端在展示和导入时是否完整使用 `description`（整段 Markdown），编辑页应能展示五维内容；若前端只展示摘要，需改前端展示逻辑。
